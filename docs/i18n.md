# Internationalisation (i18n)

Gastro doesn't ship an i18n library. It ships
[`WithRequestFuncs`](helpers.md#request-aware-helpers-withrequestfuncs)
— the per-request template-helper hook — and stays out of your way.
Adopters bring [gotext](https://pkg.go.dev/github.com/leonelquinteros/gotext),
[go-i18n](https://pkg.go.dev/github.com/nicksnyder/go-i18n/v2), or a
hand-rolled PO loader; Gastro provides the seam where they plug in.

This page walks through the full recipe end-to-end, using the bundled
`examples/i18n/` app as the worked example. The whole thing is ≈400
lines of code; the Gastro-facing wiring is ≈15 of them.

## The shape

Three things compose:

1. **An i18n library** — your choice. Exposes a per-locale localizer
   with `T(msgid)`, `TN(singular, plural, n)`, and `TC(ctx, msgid)`
   (or whatever methods your library prefers).
2. **Locale-detection middleware** — picks the right localizer for
   each request and attaches it to `r.Context()`.
3. **A `WithRequestFuncs` binder** — pulls the localizer out of the
   context and exposes its methods as template helpers.

That's it. Three pieces. The first is ≈95% of the code (and adopter
choice); the third is the ≈15-line glue.

## The walkthrough

This section walks through the canonical wiring against the bundled
`examples/i18n/` app. The full source is at `examples/i18n/` — code
snippets here are excerpts.

### 1. Pick (or write) an i18n library

`examples/i18n/` uses a tiny hand-rolled PO loader so the recipe
stays self-contained. The shape that matters:

```go
type Localizer interface {
    T(msgid string) string
    TN(singular, plural string, n int) string
    TC(ctx, msgid string) string
}
```

For a real app, swap in `gotext`:

```go
import "github.com/leonelquinteros/gotext"

// gotext.Po already has T, TN, TC methods.
```

Or `go-i18n` (the wrapping is a few lines):

```go
import goi18n "github.com/nicksnyder/go-i18n/v2/i18n"

type Localizer struct{ b *goi18n.Localizer }
func (l *Localizer) T(msgid string) string {
    s, _ := l.b.Localize(&goi18n.LocalizeConfig{MessageID: msgid})
    return s
}
// ... etc
```

The recipe doesn't care which.

### 2. Embed the catalogues

PO files are just text. Embed them at compile time:

```go
//go:embed i18n/*.po
var poFS embed.FS
```

Layout convention:

```
your-project/
├── main.go
└── i18n/
    ├── en.po       # source language catalogue
    ├── da.po
    └── de.po
```

### 3. Write the locale-detection middleware

The middleware is plain stdlib `http.Handler`-shaped — nothing
Gastro-specific:

```go
func Middleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        loc := pickLocale(r)            // 1. path prefix /da/...
                                        // 2. gastro_locale cookie
                                        // 3. Accept-Language header
                                        // 4. fallback
        ctx := context.WithValue(r.Context(), ctxKey{}, loc)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}

func FromCtx(ctx context.Context) *Localizer {
    if l, ok := ctx.Value(ctxKey{}).(*Localizer); ok {
        return l
    }
    return nullLocalizer()  // probe-safe — see contract below
}
```

**Probe-safe FromCtx is mandatory.** At `gastro.New()` time, Gastro
probes each `WithRequestFuncs` binder with a synthetic request whose
context carries no adopter-installed values. If your `FromCtx` panics
on a missing localizer, the probe panics, and `New()` panics with it.
Return a no-op localizer (one that echoes msgids back) for the empty-
context case. See [`docs/helpers.md` §"The binder contract"](helpers.md#the-binder-contract)
for the full rules.

### 4. Wire `WithRequestFuncs`

```go
router := gastro.New(
    gastro.WithMiddleware("/{path...}", cat.Middleware),
    gastro.WithRequestFuncs(func(r *http.Request) template.FuncMap {
        l := i18n.FromCtx(r.Context())
        return template.FuncMap{
            "t":  l.T,
            "tn": l.TN,
            "tc": l.TC,
        }
    }),
)
```

The binder runs once per request. The `template.FuncMap` it returns
holds **method values** — function values that capture the localizer
they're bound to. When templates call `{{ t "Welcome" }}`, the closure
resolves against the right locale because each request gets its own
binder invocation.

### 5. Use the helpers in templates

```gastro
---
import Layout "components/layout.gastro"
Items := []string{"apple", "banana", "cherry"}
Count := len(Items)
---
{{ wrap Layout (dict "Title" "Home") }}
    <h1>{{ t "Welcome" }}</h1>
    <p>{{ printf (tn "%d item" "%d items" .Count) .Count }}</p>
    <button>{{ tc "button" "Open" }}</button>
{{ end }}
```

The plural pattern follows gettext's convention: `tn` returns the
**localised format string**, and `printf` substitutes the count. Same
in every gettext library.

Helpers work inside components too — including the `wrap` block's
parent component (e.g. `Layout`), so a shared layout that calls
`{{ t "Home" }}` for nav text resolves to the right locale on every
page render. See `examples/i18n/components/layout.gastro`.

## Detection strategies

The bundled middleware tries four strategies in priority order. Adopt
the ones that fit your app; drop the rest.

### Path prefix (`/da/...`, `/de/...`)

Best for SEO. Each locale gets its own URL space. If you want the
auto-route table to mirror, write `pages/[lang]/index.gastro`-style
files; otherwise (as in `examples/i18n/`) let the middleware pluck the
prefix off the path and have one set of pages serve every locale.

### Cookie

Set a `gastro_locale` cookie on first visit (e.g. from a "language"
switcher) so returning users land in the right locale even without a
path prefix. Persists across browser sessions.

### `Accept-Language` header

The browser-supplied default. Good for first-visit fallback. The
example does a simple first-match parse; for quality-weight handling
use `golang.org/x/text/language.MatchStrings`.

### Fallback locale

When nothing else matches, ship the source-language catalogue.

## Translation workflow

`xgettext` (part of GNU gettext) extracts strings from your source.
It doesn't natively parse `.gastro` files, but they're plain text:

```sh
xgettext \
    --keyword=t:1 \
    --keyword=tn:1,2 \
    --keyword=tc:1c,2 \
    --from-code=UTF-8 \
    --output=i18n/messages.pot \
    pages/*.gastro components/*.gastro
```

Update an existing translation:

```sh
msgmerge --update i18n/da.po i18n/messages.pot
```

In dev mode, run `gastro dev --watch "i18n/*.po"` to trigger a restart
on every PO edit. Without `--watch`, the embedded catalogue stays
stale until the next manual rebuild. See [`docs/dev-mode.md`
§"Watching extra files"](dev-mode.md#watching-extra-files).

## Plural rules

The example's bundled PO loader uses a trivial `n==1 → singular, else
plural` split. That's correct for English, German, Danish, and a
handful of others, but wrong for languages with more plural forms
(Russian: 3 forms; Polish: 3; Arabic: 6; Welsh: 4).

For CLDR-correct plural rules, use `gotext`:

```go
import "github.com/leonelquinteros/gotext"

po := gotext.NewPo()
po.Parse(rawPOBytes)
// gotext.Po implements T, TN, TC with CLDR plural rules.
```

The `WithRequestFuncs` binder shape stays identical; only the
Localizer implementation changes.

## Alternative: per-page-frontmatter translation

If you only need translation in a handful of places and don't want to
register binders globally, you can skip `WithRequestFuncs` entirely:

```gastro
---
import Layout "components/layout.gastro"
Welcome := i18n.FromCtx(r.Context()).T("Welcome")
---
{{ wrap Layout (dict "Title" "Home") }}
    <h1>{{ .Welcome }}</h1>
{{ end }}
```

This is the "Tier A" pattern from the
[`WithRequestFuncs` design notes](helpers.md#request-aware-helpers-withrequestfuncs).
Works fine for content-light pages. For content-heavy templates, the
ergonomic cost (every translation needs a frontmatter var) is what
`WithRequestFuncs` is for.

## What about route-mirroring?

`examples/i18n/` doesn't auto-generate `pages/[lang]/index.gastro` —
the locale is picked at request time and the same files serve every
locale. That's simpler but suboptimal for SEO (search engines see one
URL per page, multiple translations).

For SEO-perfect setups, write `pages/[lang]/index.gastro` files
explicitly and use `r.PathValue("lang")` in frontmatter to pin the
locale. Gastro doesn't auto-mirror routes because the right strategy
depends on your URL design (subdomain? subdirectory? query
parameter?), and one-size-fits-all would lock you in.
