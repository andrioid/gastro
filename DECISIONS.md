new page `examples/gastro/pages/docs/getting-started-library.gastro` renders the new doc with prev/next navigation. **(b) `gastro watch` subcommand.** `--run COMMAND` (required, single), `--build COMMAND` (optional, repeatable for multi-step pipelines like tailwind+go-build), `--exclude PATH` (repeatable, on top of the always-on hardcoded set: `.gastro/`, `vendor/`, `node_modules/`, `.git/`, `tmp/`), `--project PATH`, `--debounce DUR`, `--quiet`, `--help`. Watches `.gastro` + `.md` (via existing markdown-deps) + `static/**` + `*.go` files anywhere under the project root (R1, R2). Smart classification (template-only edit ≠ binary restart) holds for `.gastro` files; all `.go` changes are restart-class because partial Go reloads aren't a thing the runtime can do safely. Cancel-restart on rapid edits (R3): each call to startApp cancels the previous build's `context.Context` via a `buildCancel` per-call. Build commands are NOT placed in their own process group (only `--run` is), so compound shell builds like `sh -c 'sleep 5; touch x'` may leak orphaned children when cancelled — acceptable per the plan's risk #2 ("user-`--build` outputs are typically Go binaries which are atomic via `go build`"); revisit if users start running long-lived/forking builds. Build-failure resilience (R4): on any `--build` non-zero exit, the previously-running `--run` process stays alive, the failure is logged to the terminal, and the failure message is written atomically (write-to-tmp + rename) to `.gastro/.build-error` so the browser can `console.warn` it via the new SSE event type (see (e) below). When the next build succeeds, the running binary is replaced and a normal reload event clears any prior warning state on the page. Process supervision (`cmd/gastro/process.go`): `--run` commands are spawned in their own process group (`SysProcAttr.Setpgid=true`) so `Stop()` can `Kill(-pgid, SIGTERM)` the entire descendant tree — closes the canonical foot-gun where `--run 'go run ./cmd/myapp'` would leak the grandchild binary and the second restart would fail with "address already in use". 5-second SIGTERM grace period before SIGKILL; not configurable in v1. Hand-rolled argv parser (no `flag.NewFlagSet`) because flag's repeatable-flag story requires a custom `flag.Value` and walking argv is less code; supports both `--run X` and `--run=X` forms plus short forms `-r`, `-b`, `-p`, `-q`. Build-output collision warning (§4a): startup substring-checks each `--build` for `-o <path>` writes into watched directories other than `tmp/` and emits a non-fatal heads-up if found. **(c) `gastro dev` rejects unknown flags** with the canonical multi-line message defined once in `cmd/gastro/watch.go` `devFlagRejectionMessage(flag)` and asserted byte-identical in tests: "`gastro dev: unknown flag <flag>` / `gastro dev takes no flags. For custom build/run commands, use:` / `  gastro watch --build '...' --run '...'`". `gastro dev` itself is unchanged in behaviour and still takes no flags by design. **(d) `writeGoFile` skip-if-equal + atomic write** (`internal/compiler/compiler.go`). Pre-fix: every `gastro generate` rewrote every file in `.gastro/` regardless of whether bytes had changed, causing spurious mod-time bumps for downstream watchers and unnecessary git diffs for teams that opted into committing the `.gastro/` tree. Fix: read the on-disk bytes first; if they match the formatted output, return without touching the file. When bytes do change, write to a sibling `.tmp` file then `os.Rename` into place — atomic on POSIX so no concurrent reader (e.g. parallel `go build`, watcher killed mid-write) ever sees a half-written file. Best-effort cleanup of the `.tmp` file on rename failure. Fixes the pre-existing issue noted in `tmp/two-stories-plan-v2.md` §9. **(e) `pkg/gastro/devreload.go` extended with a typed event channel.** Pre-fix: `Subscribe()` returned `<-chan struct{}` and `HandleSSE` always emitted `event: reload` — no payload, no event differentiation. Post-fix: a small `reloadEvent struct{Kind, Data string}` flows through subscribe/broadcast; `HandleSSE` emits either `event: reload` (data "reload", wire-format unchanged for backwards compat with any in-the-wild client scripts) or `event: build-error` (data is the failure message). Polling watchSignal now checks `.gastro/.reload` first (a successful reload makes any stale `.build-error` moot and removes it without broadcasting) then `.gastro/.build-error`. Client-side `reloadScript` gains one listener: on `build-error`, log via `console.warn("gastro: build failed — previous version still serving\n" + ev.data)`. A visible banner UI is explicitly **deferred** to a follow-up (plan §10) — v1 ships only the transport layer plus `console.warn`, so the banner work becomes UI-only with no protocol changes. The Subscribe channel buffer was bumped from 1 to 4 because build-error events can closely follow reload events and a strict cap-1 channel would drop the build-error if a reload was already queued. Existing in-tree callers (`internal/compiler/compiler.go`'s generated `routes.go` calling `DevReloader.Start/HandleSSE/Middleware/Broadcast`) are unaffected because Subscribe() is only used inside HandleSSE itself. **(f) `internal/devloop` package** extracted from `runDev`'s inline polling loop. `Run(ctx, Config{Generate, OnRestart, OnReload, WatchGoFiles, ExtraExcludes, ...})` is shared between `gastro dev` (WatchGoFiles=false, framework binary build via inline `startApp`) and `gastro watch` (WatchGoFiles=true, user-supplied build/run via `runWatchLoop`). The plan called for OnRestart to be "called once at startup, before the watcher starts processing events" but the literal implementation (`Generate → OnRestart → seed → watch`) raced with fast callers (integration tests editing files immediately after observing OnRestart): the seed could capture post-edit state and silently misclassify the change as `SectionUnknown` (→ `ChangeRestart` → "frontmatter" label, which is the symptom that surfaced the race). Re-ordered to `Generate → seed → OnRestart → watch` so callers waiting on the initial OnRestart can trust the seed is complete; documented inline in `internal/devloop/devloop.go`. The plan's contract ("called once at startup") is preserved and arguably strengthened. `writeReloadSignal()` stays in `cmd/gastro/main.go` and is injected via `Config.OnReload`, keeping `internal/devloop` ignorant of project-root path resolution. Empty-`pages`/`components` warnings (Phase 3) emit on startup via a swappable `stderrSink` so tests can capture without `os.Stderr` juggling. **(g) Explicit non-decisions** that the plan locked in: no pure-watch mode (`gastro watch` always requires `--run`); reserved for the existing `watchexec` recipe in `docs/dev-mode.md` until an adopter asks. No flags ever on `gastro dev` (the scaffold default is the only thing it knows; configuration lives on `gastro watch`). No Windows support in `gastro watch` v1 (process groups + `sh`-style splitting differ on Windows; the relevant `SysProcAttr.Setpgid` field doesn't exist there). No visible browser banner UI in v1 (transport ships, UI is a follow-up so it becomes pure CSS/JS work with no protocol changes). No third-party runner dependency, ever (we built a focused, gastro-aware subset of `air` precisely so users don't have to install `air`/`wgo`/`watchexec`). No mode-specific runtime APIs (e.g. `gastro.NewLibrary(...)` vs `gastro.NewFramework(...)`) — the runtime is identical between modes, the documentation framing is purely documentary. **(h) One new approved third-party dependency: `github.com/google/shlex`** (~150 LOC, BSD-3, zero transitive deps). - **2026-05-02** (m+git@andri.dk) Wave 5 closure — Q4 audit drops A2; frictions plan archived. The last open item in `docs/history/frictions-plan.md` was Wave 5 / A2 ("point `//go:embed` at source `components/` and `pages/` directly instead of `.gastro/templates/` copies"), gated on Q4 ("does this change `gastro dev` path resolution?"). The audit revealed Q4 was the wrong question: A2 itself is infeasible. The `.gastro/templates/*.html` files are not copies of source `.gastro` files — they are the **transformed template body** (frontmatter stripped via `parser.Parse`, `{{ markdown "…" }}` directives expanded by `codegen.ProcessMarkdownDirectives`, `{{ wrap Foo (…) }}` and bare component calls rewritten to compiled forms like `{{ Foo (dict "Children" (__gastro_render_children "layout_children_0" .)) }}` plus generated `{{define}}` blocks via `codegen.TransformTemplate`, filenames flattened from `pages/index.gastro` to `pages_index.html`). Pointing `//go:embed` at source directories would embed raw `.gastro` files that `html/template.Parse` cannot consume — frontmatter `---` markers aren't valid template syntax, `{{ wrap }}` is not in the FuncMap, and the dict-rewriting hasn't run. Recovering the parseable shape at runtime would require running the full codegen pipeline on every dev-mode request, which contradicts the codegen-everything model and adds a parse step to every page load. The originally-stated benefit ("cuts the embed half of the committed tree") is also obsolete: `.gastro/` is gitignored by default at the repo root and in the scaffold, `git ls-files | grep '\.gastro/'` returns zero tracked files anywhere in this repo, and `docs/design.md` §1084 ("`gastro check` for committed `.gastro/`") explicitly notes commit-the-tree is a downstream choice rather than the default. Three potential reframings were considered and rejected: embedding `.gastro` source plus runtime transform (rejects the codegen model), generating `.html` next to source files (pollutes source dirs and breaks the generated/source separation `gastro check` depends on), and generating to a user cache dir then embedding from there (`//go:embed` paths cannot escape the module). Resolution: A2 dropped, Q4 marked resolved with a pointer to the audit findings, Wave 5 closed empty. Per-project remedy for downstream commit-the-tree teams: add `.gastro/templates/` to their own `.gitignore` (the Go handler files in `.gastro/*.go` remain the reviewable surface; `gastro check` continues to catch drift because it regenerates and byte-compares the full tree). Documented this remedy in a new `docs/deployment.md` §"Trimming the committed tree" subsection. **Plan archival:** with all five waves and Track B closed, `plans/frictions-plan.md` was moved to `docs/history/frictions-plan.md` together with the source `frictions.md` audit (→ `docs/history/frictions.md`) and the companion `frictions-mode-split.html` report. The empty `plans/` directory was removed. Live code comments and doc references that pointed at `plans/frictions-plan.md` were updated to `docs/history/frictions-plan.md` (24 sites across `pkg/`, `internal/`, `examples/`, and `docs/`); historical DECISIONS.md entries were left alone because they're a dated log of paths-as-of-that-date, and rewriting them would falsify the record. - **2026-05-02** (m+git@andri.dk) Wave 4 — framework-mode additives (C2 + C4). Implements both items from `plans/frictions-plan.md` §3 Wave 4 in one session: `WithMiddleware` (route middleware composition, C2) and `WithErrorHandler` (page render error contract, C4). Both are pure additions to the `gastro.New(opts...)` surface; rule 2 (no new authoring concepts in frontmatter or templates) is preserved. Three open design questions for C2 were resolved in-session before any code was written: **Q3** — when both `WithOverride` and `WithMiddleware` target the same route, middleware wraps the override (the override resolves first, then matching middleware wraps the result, applied via a per-route `applyMiddleware` closure in the generated mux build loop). **Q6** — patterns use Go's stdlib `http.ServeMux` syntax (`{slug}` segment, `{slug...}` trailing catch-all, `/{path...}` as the canonical bare catch-all); rejected the originally-planned `*` sentinel because reusing the same syntax that gastro auto-routes already use means no parallel pattern dialect to learn. Validation uses a throwaway-mux probe (`gastroRuntime.PatternMatchesAnyRoute` in `pkg/gastro/middleware.go`): register the candidate pattern in a throwaway `http.ServeMux` with a sentinel handler, synthesise a concrete URL for each known auto-route by replacing `{name}` and `{name...}` segments with literal placeholders, dispatch each through the throwaway mux, and check whether the sentinel ever wins — if not, the pattern matches no auto-route and `New()` panics with the same wording as `WithOverride`. Reuses Go's pattern semantics 1:1, so any future change to stdlib pattern matching is automatically inherited. **Q7** — patterns are path-only, mirroring `WithOverride`; method-specific middleware (CSRF-on-POST, method-aware logging) branches on `r.Method` internally. Rejected the spike's original method-scoped form because the asymmetry where `WithOverride("POST /counter", ...)` rejects but `WithMiddleware("POST /counter", ...)` accepts would surface as a confusing user-facing inconsistency. Audit of `internal/compiler/compiler.go` route-pattern validation concluded `WithOverride`'s exact-match validation (lines ~744–758 of the routes template) should stay untouched: C2's validation diverges fundamentally (throwaway-mux probe vs map lookup), so it lives as a sibling block, not a refactor. Throwaway-mux probe lives in `pkg/gastro/middleware.go` for `-race`-clean unit-testability; the codegen template stays thin. **C2 implementation:** new `MiddlewareFunc` type alias and `MiddlewareApplies(middlewarePattern, routePattern string)` helper in `pkg/gastro/middleware.go`; new `WithMiddleware(pattern, fn)` option in `internal/compiler/compiler.go`; codegen template grows a `middleware []middlewareEntry` field on `config`, a validation block that calls `gastroRuntime.ValidateMiddlewarePattern` for each registered pattern, and a per-route `applyMiddleware(route, h)` closure used by both the static-asset and auto-route branches of the mux build loop. Walks `cfg.middleware` in reverse so registration-order composes outermost-to-innermost: the first registered middleware ends up outermost on the request, last on the response. Tests: 4 unit tests in `pkg/gastro/middleware_test.go` cover probe semantics across exact, parametrised, and wildcard patterns plus error wording and composition shape; 5 integration tests in `internal/compiler/middleware_integration_test.go` build a temp project with `/`, `/admin/users`, `/admin/settings` and verify exact-match scoping, wildcard subtree scoping, registration-order composition (asserted via ordered response header values), middleware-wraps-override (Q3), and the panic for unknown patterns.
